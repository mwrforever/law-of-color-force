# 《色力法则》（Color Power Rules）技术方案

> 文档性质：技术方案制定阶段（只做技术方向 / 架构对比 / 推荐，**不含任何游戏业务代码，不含 GDD**）。
> 适用范围：微信生态 3D 物理动作解谜 / 派对游戏。
> 作者：工程负责人 程基岩（engineering-lead）
> 版本：v1.0（关键技术决策已于 2026-07-15 经用户确认定稿）

---

## 0. 平台定论：微信**小游戏** vs 微信**小程序**

### 0.1 结论（一句话）
《色力法则》是 3D 物理实时改造 + 2–4 人低延迟派对游戏，**技术载体必须是「微信小游戏（mini game）」，不能用普通「微信小程序（mini program）」**。用户口头说的"微信小程序游戏"是口语误称，方案以"微信小游戏"为准。

### 0.2 为什么不是小程序（可行性论证）

| 维度 | 微信小程序（mini program） | 微信小游戏（mini game） | 对本项目的影响 |
|---|---|---|---|
| 运行环境 | 内嵌 WebView（iOS=WKWebView / Android=XWeb WebView） | 微信自研游戏运行时（Android=V8，iOS=JSCore）+ WebGL | 小游戏有独立优化过的渲染/脚本循环，小程序没有 |
| 渲染 | `canvas type="webgl"` 支持有限、内存/性能受限 | 一等公民 WebGL（Metal / OpenGL ES 后端），`requestAnimationFrame` 游戏主循环 | 3D 物理 60fps 必须小游戏 |
| 入口与构建 | `app.json` / 页面式 | `game.js` + `game.json`，游戏式 | 架构模型完全不同 |
| 包体 | 主包 2MB，分包总 20MB（更紧） | 主包 4MB，单分包 4MB，总 20MB；**资源可远程加载不计入包体** | 小游戏余量更大 |
| 游戏专属 API | ❌ 无 | ✅ 实时语音/房间、录屏 `getGameRecorderManager`、开放数据域排行榜 `createOpenDataContext`、重力感应、好友关系链 | 派对语音、排行榜、社交裂变都依赖这些 |
| 多人实时语音 | ❌ 不支持微信实时语音/房间 | ✅ 微信实时语音服务 / 房间管理能力 | "色彩大乱斗"语音派对必须有 |
| 生产级 3D 物理 | 仅能跑 demo 级 three.js webgl，生产不可行 | 可行（见后文引擎/物理方案） | 决定性差异 |

**可行性结论**：小程序**技术上上限只能跑 three.js WebGL 演示**，在 (a) 持续 60fps 3D 渲染、(b) 大批量刚体物理、(c) 实时语音派对、(d) 录屏/排行榜社交 四个维度全部不达标。**若强行用小程序，需砍掉 3D 物理与派对语音，等于重做产品，不推荐。**

### 0.3 走「微信小游戏」需要做的改造清单

1. **运行时适配**：用引擎官方「微信小游戏构建目标」（Cocos / Laya 自带）；若用 Three.js / Babylon.js，需引入 `weapp-adapter`（微信小游戏适配器，把 `wx.*` 桥接成 `window/document/canvas` 等 DOM-like API）。
2. **入口与生命周期**：`game.js` 启动 + `game.json` 配置（设备方向、网络超时、分包）；放弃小程序的页面路由模型。
3. **包体治理（核心）**：主包 4MB 只放**引擎核心 + 首屏必要资源**；其余走**分包（≤4MB/个）** + **远程资源（CDN / 微信云存储 `wx.downloadFile`，不计入 20MB 上限）**。WASM、贴图、模型、音频尽量远程化或分包。
4. **API 替换（无 DOM）**：
   - 输入：`wx.onTouchStart/Move/End`（多点触控支持喷枪涂抹）
   - 图片：`wx.createImage()`；音频：`wx.createInnerAudioContext()`
   - 文件：`wx.getFileSystemManager()`；远程资源：`wx.downloadFile()`
   - 录屏：`wx.getGameRecorderManager()`；语音/房间：微信实时语音服务
   - 排行榜：`wx.createOpenDataContext()`（独立渲染线程，防作弊改分）
5. **性能与审核**：首屏资源精简、分包预下载、贴图压缩（ktx2 / pngquant）、模型 Draco 压缩；小游戏需**游戏版号 / 类目资质**。

## 1. 3D 物理引擎对比与推荐

### 1.1 需求映射（颜色 → 物理参数）
核心机制是"运行时动态修改**单个物体**物理参数"，先定义映射模型（所有引擎都必须能落到这套 profile）：

| 颜色 | 物理语义 | 物理参数改造 |
|---|---|---|
| 🔴 红 | 高重力 | `gravityScale` 高（额外向下力） |
| 🔵 蓝 | 低/零重力（漂浮） | `gravityScale≈0` + 微浮力 |
| 🟡 黄 | 高弹性 | `restitution` 高 |
| 🟣 紫（红+蓝） | 固化/定身 | body 切 `STATIC/KINEMATIC`（冻结） |
| 🟠 橙（红+黄） | 重力+弹性 | 高 `gravityScale` + 高 `restitution` |
| 🟢 绿（蓝+黄） | 低重力+弹性 | 低 `gravityScale` + 高 `restitution` |

**关键能力要求**：① 单物体 `gravityScale`；② `restitution`；③ 运行中 `DYNAMIC ↔ STATIC` 切换（定身）。

### 1.2 候选对比

#### A. cannon-es（纯 JS，cannon.js 续作）
- **定位**：最轻量的纯 JS 3D 物理库，MIT。
- **适用场景**：中小规模刚体（几十~百级）、对包体极敏感、需要直接改 JS 对象的休闲游戏。
- **《色力》优劣势**：
  - ✅ 包体极小（min ~200KB，gzip ~70KB），纯 JS 无 WASM，4MB 主包无压力。
  - ✅ 参数都是 JS 对象，改 `body.mass` / `body.type` / `restitution` 极直观。
  - ⚠️ **无原生 `gravityScale`**（cannon-es 的 `Body` 不内置逐体重力缩放）→ 需每 tick `applyForce` 或自扩展；定身用 `body.type = Body.STATIC` + `mass=0` 可行。
  - ❌ 求解器是顺序冲量，大规模刚体（数百+）性能弱；无软体/高级约束。
- **包体/性能量化**：~200KB JS；移动端舒适区 ~50–150 动态刚体 @60fps；>300 明显掉帧。

#### B. ammo.js（Bullet 的 Emscripten WASM 移植）
- **定位**：业界标准 Bullet 物理的 WASM 版，能力最全。
- **适用场景**：需要 Bullet 级稳定性、软体、复杂约束、大刚体量。
- **《色力》优劣势**：
  - ✅ 功能最全（刚体/软体/约束），`btRigidBody` 支持逐体重力、restitution、定身（`setMass(0)`+停激活）。
  - ❌ **WASM 体积大**（wasm 构建 ~600KB–1.5MB，视特性裁剪；asm.js 弃用），直接吃掉主包/分包预算。
  - ❌ Emscripten 自动生成的 API 冗长难用（`new Ammo.btVector3(...)` 需手动 `destroy` 防内存泄漏），与 JS 层耦合与调试成本高。
- **包体/性能量化**：wasm ~0.6–1.5MB（需实测裁剪后体积）；可支撑数百~千级刚体。

#### C. rapier（Rust→WASM，dimforge）
- **定位**：现代 Rust 物理引擎的 WASM 版，**默认确定性**（帧同步友好）。
- **适用场景**：需要高性能 + **跨平台确定性** + 干净 API 的项目（尤其帧同步多人）。
- **《色力》优劣势**：
  - ✅ **性能强**（宽相/窄相先进），千级刚体 @60fps；API 干净：`setGravityScale()` / `setRestitution()` / `setBodyType(Static)` 全部原生 → **完美契合"单物体动态改参"**。
  - ✅ **确定性**是帧同步多人的稀缺优势。
  - ⚠️ WASM 体积（rapier3d 约 1.5–2.5MB，compat 版可动态加载）；需异步初始化（先 load wasm）。
  - ⚠️ 生态比 cannon 小，中文资料少。
- **包体/性能量化**：wasm ~1.5–2.5MB（以官方 release 为准，建议实测）；刚体千级。

#### D. 引擎自带物理（Cocos 内置 cannon / Cocos PhysX 插件 / Laya PhysX 绑定）
- **定位**：引擎内建，与渲染/场景图深度集成，免手动桥接。
- **适用场景**：用 Cocos / Laya 做渲染时直接复用其物理系统。
- **《色力》优劣势**：
  - ✅ 与编辑器、碰撞回调、动画系统打通，开发效率最高。
  - ✅ Cocos 内置 physics 基于 cannon（轻，零额外包体）；Cocos 另有 **PhysX WASM 插件**（~2–4MB，走分包，能力对标 Bullet/PhysX，含逐体重力、restitution、kinematic 定身）。
  - ⚠️ 内置 cannon 受 cannon 性能上限约束；PhysX 插件体积需分包消化（体积以官方为准）。
- **包体/性能量化**：内置 cannon ≈ 与 A 同；PhysX 插件 wasm ~2–4MB（需实测）。

### 1.3 推荐

> **推荐：优先「引擎自带物理」路线——Cocos Creator 3D 内置 cannon 起步（零额外包体），若派对模式确定走帧同步或大刚体场景，升级到 Cocos PhysX WASM 插件（分包承载）。若团队选择 Three.js/Babylon.js 自研渲染，则物理选 rapier（WASM 分包）。**

**推荐理由**：
1. **单物体动态改参是核心**，cannon / PhysX / rapier 都能落（cannon 用 `applyForce`+类型切换，PhysX/rapier 有原生 API）。
2. **包体是硬约束**：cannon 零额外包体最优；rapier / PhysX 的 WASM 用**分包 + 远程加载**消化，不占主包。
3. **确定性**：若派对走帧同步，rapier / PhysX 优于 cannon（cannon 浮点不确定）。但若走状态同步（见第 3 节推荐），确定性要求降低，cannon 即可。

## 2. 渲染方案对比与推荐

### 2.1 候选对比

#### A. Three.js（纯渲染库，JS）
- **定位**：最流行的 WebGL 渲染库，生态最大，可树摇。
- **适用场景**：想要最大控制力、自定义渲染管线、轻量首包。
- **《色力》优劣势**：
  - ✅ 核心 ~150KB（full ~600KB gzip），可树摇，主包友好；社区资源最多。
  - ✅ 波普/橡皮泥质感靠**自定义 ShaderMaterial**（卡通/Toon 渐变 + 顶点色 + 轮廓线 + Bloom/色差后处理）完全可控。
  - ❌ **无编辑器**：场景、光照、相机全靠代码搭，3D 关卡美术迭代慢（对解谜/派对大量关卡是痛点）。
  - ❌ 小游戏需 `weapp-adapter` 适配；与物理引擎手动桥接（无内建物理插件体系）。
- **包体/性能**：~150–600KB；性能取决于用法，纯渲染层很轻。

#### B. Babylon.js（全功能引擎，JS/TS）
- **定位**：企业级完整 3D 引擎，自带物理插件体系、节点材质编辑器、后处理。
- **适用场景**：想要"开箱即用"的完整引擎能力且接受较大包体。
- **《色力》优劣势**：
  - ✅ 内建 **Physics Plugin**（Cannon/Ammo/Rapier/Oimo 任选）；**Node Material 可视化着色** + 丰富后处理（Bloom/描边）契合波普风。
  - ✅ 有官方/社区 小游戏适配经验。
  - ❌ 包体大（min ~3MB+），**主包 4MB 放不下 → 必须分包/远程化引擎本体**，首屏加载链变长。
- **包体/性能**：~3MB+，需分包；性能强（WebGL2）。

#### C. Cocos Creator 3D（国产全功能引擎）
- **定位**：国产 3D 引擎，**微信小游戏一等公民**（官方构建目标 + 分包 + 开放数据域 + 资源管线全内置）。
- **适用场景**：中文团队、要编辑器提效、要小游戏一键导出。
- **《色力》优劣势**：
  - ✅ **原生小游戏支持**（构建直接出小游戏包，自动处理 adapter/分包/资源压缩），运维成本最低。
  - ✅ **可视化编辑器**：场景/光照/动画/粒子拖拽，关卡迭代快（解谜+派对大量关卡的关键）。
  - ✅ 内建 **Toon 卡通材质 + 后处理（Bloom/描边）**，波普/橡皮泥质感可直接做；内建物理（cannon / PhysX 插件）。
  - ⚠️ 引擎本体仍占包体（可用分包/远程化缓解）；社区国际化弱于 Three/Babylon。
- **包体/性能**：引擎本体需分包；性能对标主流，优化成熟。

#### D. LayaAir 3D（国产引擎）
- **定位**：国产轻量 3D 引擎，小游戏支持好，有 PhysX 绑定。
- **适用场景**：追求更轻量、偏 2D/轻 3D 混搭的中文项目。
- **《色力》优劣势**：
  - ✅ 小游戏支持成熟，包体比 Cocos 更轻；PhysX 绑定可用。
  - ⚠️ 3D 生态/编辑器成熟度略逊 Cocos；波普质感需更多自建。
- **包体/性能**：较轻；中等 3D 场景胜任。

### 2.2 波普 / 橡皮泥质感实现要点（与引擎无关的设计）
- **卡通着色（Toon / Cel）**：离散光照阶 + 色块，高饱和纯色。Cocos/Babylon 有现成 Toon 材质；Three 用 `MeshToonMaterial` + gradientMap。
- **橡皮泥质感**：次表面近似（暖色透感）+ 略高 roughness + 圆角法线 + 轻微顶点扰动；可用 **顶点色** 承载玩家喷涂的局部染色。
- **色彩碰撞视觉**：后处理 **Bloom（辉光）+ 轻微色差/饱和度增强 + 描边（Outline）**；喷涂瞬间用粒子 + 泼溅贴花。
- **动态喷涂**：喷枪 = 射线拾取 + 在材质上局部改写顶点色/贴图区域（运行时生成/更新 `Texture` 或顶点色缓冲）。

### 2.3 推荐

> **推荐渲染方案：Cocos Creator 3D。**
> 替代方案：若团队坚持"引擎无关 + 最大控制"，选 **Three.js + 自研管线 + weapp-adapter**（包体最省，但关卡编辑/物理桥接成本转嫁到人力）；**Babylon.js** 适合能接受引擎分包、想要完整物理插件与节点材质的中大型团队。

**推荐理由**：① 小游戏原生构建，省去 adapter/分包/资源管线自研；② 编辑器对"大量关卡 + 实时喷涂调试"提效显著；③ 内建 Toon + 后处理 + 物理，最贴合波普风与"色彩改物理"机制；④ 中文团队支持与案例最丰富。

## 3. 多人实时同步对比与推荐

### 3.1 小游戏网络限制（前提）
- 小游戏网络走微信通道，**必须走 `wx.connectSocket`（WebSocket）或微信云开发**，不能直接裸 TCP（iOS 限制）；
- 域名需**配置合法域名白名单**（或开发期不校验）；
- 实时语音走微信**实时语音服务 / 房间** API（不占游戏 WebSocket 带宽）；
- 单局 2–4 人，**端到端延迟目标 < 100–200ms**（同地域微信通道通常可达）。

### 3.2 帧同步 vs 状态同步（先定模式）

| 模式 | 原理 | 优点 | 缺点 | 适配度 |
|---|---|---|---|---|
| **帧同步** | 服务器只转发输入，各端确定性模拟 | 带宽极低、服务器极便宜 | 需**确定性物理**（rapier/PhysX，非 cannon）；浮点/锁步难；卡顿传染；**易作弊（改内存）** | 中（竞技硬核合适，但作弊风险高） |
| **状态同步** | 服务器权威，客户端发输入、收状态 | 抗作弊、稳健、易调试 | 带宽/服务器成本较高；需服务器跑逻辑 | **高**（派对休闲 + 皇冠争夺怕作弊） |

> 派对模式含**皇冠争夺/竞速**等"有输赢"的对抗 → **作弊是真实风险**，倾向**状态同步（或 host 权威）**。帧同步仅在对延迟极致敏感且接受反作弊弱时考虑。

### 3.3 候选方案对比

#### A. 微信小游戏「房间/实时语音」+ 自研帧同步
- **定位**：用微信做匹配/语音，自己写帧同步逻辑。
- **适用场景**：竞技硬核、极低成本、能接受反作弊弱。
- **《色力》优劣势**：✅ 微信房间/语音现成；服务器极省。❌ 需确定性物理（锁死 rapier/PhysX）；2–4 人派对对作弊零容忍时不合适；自研锁步复杂度高。

#### B. 微信云开发（CloudBase）实时数据库
- **定位**：云函数 + 实时数据监听做房间状态。
- **适用场景**：极简后端、低并发、对实时性要求不高的同步。
- **《色力》优劣势**：✅ 免运维、上手快、天然小游戏集成。❌ 实时监听是**监听推送（~100–300ms 级）**，达不到动作派对 <200ms 手感；高频状态更新成本高、易触发限频。**不适合作为动作同步主通道**。

#### C. 第三方帧同步 / 语音 SDK（Matchvs / GoOnline / 即构 / 声网）
- **定位**：现成帧同步 + 语音一体化。
- **适用场景**：不想自研网络层、要快速上线。
- **《色力》优劣势**：✅ 接入快、有帧同步+语音；✅ 声网/即构语音质量高。❌ 按量付费（2–4 人派对长期成本需算账）；绑定第三方、数据出境/合规需评估；帧同步仍受确定性约束。

#### D. 自建 WebSocket 信令 + 状态同步（推荐底座）
- **定位**：自己写 WebSocket 服务做房间 + 状态同步，权威端在服务器或 host 客户端。
- **适用场景**：要低延迟、抗作弊、可控成本。
- **《色力》优劣势**：
  - ✅ **延迟可控**（WebSocket 双向，<200ms 可达）；状态同步抗作弊；逻辑完全可控。
  - ✅ 部署在 **CloudBase 云托管 / 云函数 WebSocket** → 仍免运维服务器，契合微信生态。
  - ✅ 语音用**微信实时语音服务**独立通道，不挤占游戏 WebSocket。
  - ⚠️ 需自研房间/匹配/状态序列化（但 2–4 人规模不大，工作量可控）；需处理弱网重连/插值。

### 3.4 推荐

> **推荐多人同步：微信实时语音/房间（匹配+语音）+ 自建 WebSocket 状态同步（部署于 CloudBase 云托管），权威端用「host 客户端权威」或「轻量云函数中继」二选一。实时数据库仅作房间列表/好友态，不作动作同步主通道。**

**推荐理由**：
1. **动作手感要低延迟** → WebSocket 而非实时数据库。
2. **派对有输赢怕作弊** → 状态同步（host 权威或服务器中继）优于帧同步。
3. **小团队免运维** → 跑在 CloudBase，不用自建机房；语音用微信原生通道。
4. **2–4 人规模** → 状态同步带宽/服务器成本可接受，无需为省带宽去啃确定性物理。

> 备选：若后期要做硬核竞技且能接受反作弊弱，可切 rapier 确定性 + 帧同步（SDK 方案 C）。

## 4. 总体推荐组合（落到具体）

| 层 | 推荐 | 备选 |
|---|---|---|
| **平台** | 微信小游戏（mini game） | —（小程序不可行） |
| **渲染** | **Cocos Creator 3D**（原生小游戏构建 + 编辑器 + Toon/后处理） | Three.js + weapp-adapter（极致控制/省包体）；Babylon.js（完整引擎分包） |
| **物理** | **Cocos 内置 cannon 起步**（零额外包体）；派对帧同步/大刚体时升 **Cocos PhysX WASM 插件**（分包） | 若自研渲染则 **rapier**（WASM 分包，确定性佳） |
| **同步** | **微信实时语音/房间 + 自建 WebSocket 状态同步（CloudBase 云托管）**，host 权威或云函数中继 | 第三方帧同步 SDK（Matchvs/声网）作后期硬核备选 |

**为什么这组最契合"色彩物理实时改造 + 2–4 人低延迟派对"**：
- **色彩改物理**：Cocos 内建物理 + 编辑器，单物体 `gravityScale/restitution/类型切换` 直接落 ColorPhysicsProfile；喷涂靠顶点色/局部贴图运行时改写，编辑器可实时调参。
- **低延迟派对**：WebSocket 状态同步 <200ms + 微信原生语音；状态同步天然抗"皇冠争夺"作弊；CloudBase 免运维。
- **包体可控**：引擎本体 + cannon 进主包/分包，PhysX/rapier 的 WASM 走分包 + 远程资源，严守 4MB 主包红线。

## 5. 已知技术风险与缓解

| # | 风险 | 缓解手段 |
|---|---|---|
| R1 | **WASM 包体超 4MB 主包**（PhysX/rapier wasm ~1.5–4MB） | 主包只放引擎核心 + 首屏；WASM/大资源走**分包 + 远程加载（CDN/云存储，不计入 20MB）**；用 compat 版动态加载 wasm |
| R2 | **iOS 小游戏 JS 引擎（JSCore）Float32/物理性能** | 控制动态刚体数量（分场景/分房间）；物理步进与渲染解耦；低端机降级（降刚体数/降帧）；用 WASM 物理（rapier/PhysX）比纯 JS cannon 在 iOS 更稳 |
| R3 | **iOS 小游戏 WebGL/特性限制** | 小游戏运行时已支持 WebGL2 + wasm；避免依赖桌面独有扩展；真机（iOS/Android）早期介入测试 |
| R4 | **确定性/同步浮点漂移**（若误用帧同步 + 非确定性物理） | 坚持状态同步（推荐）；若用帧同步必须 rapier/PhysX + 定点/锁步，并做跨端一致性测试 |
| R5 | **美术资源体积（高饱波普贴图/模型）** | ktx2 / pngquant 压缩、Draco 模型压缩、按需分包预下载、远程资源 |
| R6 | **派对模式作弊（改内存抢皇冠）** | 状态同步 + 服务器/host 权威校验；关键分值走开放数据域或服务器；输入校验 |
| R7 | **小游戏审核 / 版号 / 类目资质** | 提前确认游戏类目与版号路径；避免未授权 IP/素材 |
| R8 | **远程资源加载链路过长导致首屏慢** | 首屏极小化 + 分包预下载 + 加载进度与弱网重试 |

## 6. 基础层架构决策记录（ADR）

> ADR 模板：决策 / 背景 / 考虑过的方案 / 最终选择 / 后果。

### ADR-001：运行平台选「微信小游戏」而非「微信小程序」
- **背景**：产品是 3D 物理实时改造 + 2–4 人语音派对，用户口语称"小程序"但技术载体不符。
- **考虑过的方案**：① 微信小程序（WebGL canvas）；② 微信小游戏。
- **最终选择**：微信小游戏（mini game）。
- **后果**：获得 WebGL 游戏运行时、实时语音/房间、录屏、开放数据域排行榜等游戏专属能力；需按 `game.js/game.json` 模型与 adapter/分包/资源管线改造；需游戏版号/类目资质。小程序路线被否决（生产级 3D 物理 + 派对语音不可行）。

### ADR-002：渲染引擎选 Cocos Creator 3D
- **背景**：需小游戏原生支持、可视化编辑器提效、波普/橡皮泥质感（Toon + 后处理）、与物理集成。
- **考虑过的方案**：Three.js（控制力最强/包体最小但无编辑器）、Babylon.js（完整但包体大需分包）、Cocos Creator 3D、LayaAir 3D。
- **最终选择**：Cocos Creator 3D（备选 Three.js 自研管线）。
- **后果**：小游戏一键构建 + 编辑器加速关卡迭代 + 内建 Toon/后处理/物理；代价是引擎本体需分包/远程化，且国际化社区弱于 Three/Babylon。若未来需引擎无关或极致包体，可切换到 Three.js 路线（成本：重建场景工具链与物理桥接）。

### ADR-003：物理引擎走「引擎自带」路线（cannon 起步，PhysX/rapier 按需升级）
- **背景**：核心机制是运行时单物体动态改参（gravityScale/restitution/定身），且受 4MB 主包约束。
- **考虑过的方案**：cannon-es（轻/无 gravityScale 原生）、ammo.js（全/包体大/API 丑）、rapier（强/确定性/WASM）、引擎自带 cannon / PhysX 插件。
- **最终选择**：Cocos 内置 cannon 起步（零额外包体）；派对帧同步或大刚体时升 Cocos PhysX WASM 插件（分包）或 rapier（自研渲染时）。
- **后果**：起步包体最优、与编辑器/碰撞回调打通；cannon 无原生 per-body gravityScale（用 `applyForce` 或类型切换补偿），且浮点非确定性（故同步走状态同步，见 ADR-004）。WASM 体积由分包/远程化消化。若后期改帧同步，则必须切到 rapier/PhysX 以保证确定性。

### ADR-004：多人同步采用「状态同步」而非「帧同步」
- **背景**：2–4 人派对含皇冠争夺/竞速等有输赢对抗，需低延迟且怕作弊；小游戏网络走 WebSocket/云开发。
- **考虑过的方案**：帧同步（微信房间+自研 / 第三方 SDK）、状态同步（自建 WebSocket + CloudBase）、实时数据库。
- **最终选择**：微信实时语音/房间（匹配+语音）+ 自建 WebSocket **状态同步**（CloudBase 云托管，host 权威或云函数中继）；实时数据库仅作房间/好友态。
- **后果**：延迟可控（<200ms）、抗作弊、逻辑可控、免运维；代价是自研房间/序列化/弱网重连（2–4 人规模可控），且状态同步带宽/服务器成本高于帧同步（可接受）。若未来做硬核竞技且接受反作弊弱，可切 rapier 确定性 + 帧同步。

### ADR-005：资源与包体采用「主包精简 + 分包 + 远程资源」三级策略
- **背景**：主包 4MB 红线，WASM/高饱波普贴图/模型易超。
- **考虑过的方案**：全打主包（不可行）、纯分包、分包+远程资源。
- **最终选择**：主包只放引擎核心+首屏必要资源；其余走分包（≤4MB/个）+ 远程资源（CDN/云存储，不计入 20MB）。
- **后果**：严守 4MB 主包，大资源不占包体上限；代价是远程加载链路需管理（预下载/进度/重试），首屏设计要极小化。

1. **派对模式同步模型**：确认走「状态同步（host 权威/云函数中继）」还是为硬核竞技预留「帧同步（需 rapier/PhysX 确定性）」？→ 直接影响物理引擎终选（cannon vs PhysX/rapier）。

## 8. 决策确认记录（2026-07-15，经用户拍板）

以下 4 项关键技术决策已由用户确认，文档定稿为 v1.0：

1. **同步模型 → 状态同步**：派对模式采用 host 客户端权威 / 轻量中继的状态同步；物理引擎据此可用 Cocos 内置 cannon 起步（无需确定性物理）。
2. **渲染引擎 → Cocos Creator 3D**：采用官方微信小游戏构建目标，复用可视化编辑器与内建 Toon / 后处理 / 物理。
3. **资源策略 → 接受远程资源加载**：主包仅放引擎核心 + 首屏，贴图 / 模型 / WASM 走分包 + CDN / 云存储（不计入 20MB 上限）。
4. **后端部署 → 自托管 WebSocket（用户偏离推荐项）**：多人中继服务由团队自建（Node / Go 等），而非 CloudBase 云托管。客户端状态同步协议不变，仅中继地址不同；需自行负责部署 / 扩缩容 / 监控。**ADR-004 的最终选择据此更新为「自托管 WebSocket 状态同步」，CloudBase 云托管降为备选。**
